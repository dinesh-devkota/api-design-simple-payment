## Q1: "Walk me through what happens, in full detail, when `POST /one-time-payment` is called."

**A — the complete request lifecycle:**

1. **TCP/HTTP** — the client opens a TCP connection to port 8080. Spring Boot's embedded Tomcat (auto-configured via `spring-boot-starter-web`) accepts the connection.

2. **Servlet / DispatcherServlet** — Spring MVC's `DispatcherServlet` receives the HTTP request and routes it to `PaymentController.oneTimePayment()` based on the path and method declared in the generated `PaymentApi` interface.

3. **Request deserialization** — Jackson's `HttpMessageConverter` reads the JSON body and constructs a `OneTimePaymentRequest` POJO. Jackson uses the `@NoArgsConstructor` + setters (or `@JsonDeserialize`) generated by Lombok.

4. **Bean Validation** — `@Valid` triggers JSR-380 validation. If `paymentAmount < 0.01` or `userId` is blank, `MethodArgumentNotValidException` is thrown **before** the method body executes. `GlobalExceptionHandler.handleValidation()` catches it and returns `400`.

5. **MDC setup** — `MDC.put("userId", request.getUserId())` — all subsequent log lines on this thread include `userId`.

6. **Idempotency check** — `IdempotencyGuard.resolve()`:
   - If no `Idempotency-Key` header: proceed directly.
   - If key present: check `RedisIdempotencyStore.find()` → `GET idempotency:{key}` in Redis.
   - Cache hit: deserialize JSON → return cached `OneTimePaymentResponse` immediately (no payment processed).
   - Cache miss: proceed to step 7.

7. **Use-case invocation** — `ProcessPaymentService.process(userId, paymentAmount)`:
   - Domain validation: `paymentAmount > 0`.
   - `AccountSpi.findById()` → `AccountAdapter.findById()` → `AccountRedisRepository.findById()` → `HGET account:{userId}` in Redis.
   - If not found: `AccountNotFoundException` → `GlobalExceptionHandler.handleAccountNotFound()` → `404`.
   - Match calculation: `MatchCalculationServiceImpl.getMatchPercentage()` + `calculateMatchAmount()`.
   - Insufficient balance check: if `payment + match > balance` → `InsufficientBalanceException` → `422`.
   - New balance: `previousBalance - (payment + match)`, scaled to 2dp.
   - Due date: `LocalDate.now(clock).plusDays(15)` → weekend-shift logic.
   - `AccountSpi.save()` → `AccountRedisRepository.save()` → `HSET account:{userId} balance {newBalance}`.
   - Structured log line emitted.
   - Returns `PaymentResult` record.

8. **Idempotency store** — if an `Idempotency-Key` was present, `RedisIdempotencyStore.store()` → `SET idempotency:{key} {json} EX 86400`.

9. **Response mapping** — `PaymentResponseMapper.toResponse()` (MapStruct-generated) maps `PaymentResult` → `OneTimePaymentResponse`.

10. **Serialization** — Jackson serializes `OneTimePaymentResponse` to JSON. `LocalDate` fields are serialized as `"2026-04-28"` (not a timestamp array) because `write-dates-as-timestamps: false` is set.

11. **MDC cleanup** — `MDC.remove("userId")` in the `finally` block.

12. **Response** — `ResponseEntity.ok(response)` → HTTP 200 with the JSON body.

## Q2: "What are the two biggest risks in this codebase for a production deployment?"

**A:**

**Risk 1: Race condition on concurrent payments for the same user.**
Two simultaneous requests both read the same balance, both calculate the same new balance, both write back — one deduction is lost. This is a financial correctness bug that would result in Paytient absorbing losses. Fix: Lua script or Redis Cluster transactions for atomic check-and-deduct.

**Risk 2: No authentication or authorisation.**
Any caller can submit a payment for any `userId`. A user who knows `user-001` (or guesses it) can drain another user's account balance to zero. Fix: OAuth 2.0 / JWT, with the `userId` extracted from the token claim — not from the request body.

## Q3: "Why do you have two Redis templates (`StringRedisTemplate` and `RedisTemplate<String, Object>`) in the same project?"

**A:** They serve different purposes:

- `StringRedisTemplate` is used by `RedisIdempotencyStore` to store manually Jackson-serialized JSON strings. The developer controls the serialization format explicitly — no `_class` field, predictable JSON, deserializable by type.

- `RedisTemplate<String, Object>` (configured in `RedisConfig`) uses `GenericJackson2JsonRedisSerializer` for values, which adds a `_class` field for polymorphic deserialization. This is available in the context but is not directly used by any adapter — the `AccountRedisRepository` (Spring Data Redis) manages its own serialization internally via `@RedisHash`.

In practice, `RedisConfig` could be removed if `AccountRedisRepository` is the only Redis access and `RedisIdempotencyStore` uses `StringRedisTemplate`. The explicit `RedisConfig` bean exists to ensure any future use of `RedisTemplate<String, Object>` gets human-readable JSON values in RedisInsight rather than binary Java serialization.

## Q4: "If this project moved to PostgreSQL, what exactly would you change?"

**A:** Only the `infra` module changes:

1. **Remove** `AccountEntity` (`@RedisHash`) — replace with a JPA `@Entity` class with `@Table`, `@Column`, etc.
2. **Remove** `AccountRedisRepository` — replace with a `JpaRepository<AccountJpaEntity, String>`.
3. **Remove** `AccountAdapter` Redis implementation — write a new `AccountJpaAdapter implements AccountSpi`.
4. **Remove** `RedisIdempotencyStore` — write a new `JdbcIdempotencyStore implements IdempotencyStoreSpi` using a database table.
5. **Update** `RedisConfig` — remove or replace with `DataSourceConfig` / JPA config.
6. **Update** `application.yml` — replace `spring.data.redis.*` with `spring.datasource.*` and `spring.jpa.*`.
7. **Update** `pom.xml` (`infra`) — replace `spring-boot-starter-data-redis` with `spring-boot-starter-data-jpa` + JDBC driver.

**Nothing changes in `domain`, `app`, or `bootstrap`.** This is the exact promise of the hexagonal architecture.

## Q5: "Why is `ProcessPaymentUseCase` an interface if there's only one implementation? Isn't that over-engineering?"

**A:** (See Q&A 14 for the full answer — the key points:)

- Any class that injects `ProcessPaymentUseCase` can be tested with a Mockito mock. If the controller injected `ProcessPaymentService` directly, tests would need all of `ProcessPaymentService`'s dependencies.
- The interface IS the contract — it documents what is guaranteed without exposing how.
- One implementation today does not mean one implementation forever — a `BatchPaymentService` implementing the same interface could process scheduled bulk payments.
- This is the standard pattern in clean architecture. The question is better asked in reverse: "why would you skip the interface?" The only valid answer is "the class is purely internal and never injected from outside its own module."

## Q6: "What does `@ConditionalOnMissingBean` guarantee, and can it ever fail?"

**A:** It guarantees that `systemClock()` only registers if no `Clock` bean already exists in the context. This creates a fallback pattern: `fixedClock` registers when `app.fixed-date` is set; `systemClock` registers otherwise.

**Can it fail?**
Yes, in subtle ways:
- **Auto-configuration ordering** — if `ClockConfig` is processed before the property is fully resolved, `@ConditionalOnProperty` might not see the value. Spring Boot's auto-configuration ordering annotations (`@AutoConfigureBefore`, `@AutoConfigureAfter`) prevent this, but custom `@Configuration` classes run in an order determined by component scan, which could theoretically vary.
- **Both conditions could be satisfied** — if someone declares another `@Bean Clock` in a different `@Configuration` class, and that class is processed *after* `ClockConfig`, both `fixedClock` (from property) and the other bean would register, causing `NoUniqueBeanDefinitionException`. `@Primary` would resolve this but adds fragility.
- **The `--debug` flag** — running with `spring.main.debug=true` prints the auto-configuration conditions report, which shows exactly why each conditional bean was or wasn't created. Essential for debugging conditional registration failures.

## Q7: "A QA engineer reports that when they run all integration tests simultaneously, they intermittently fail with connection refused on port 6381. What's the problem?"

**A:** Port 6381 is hardcoded in `RedisTestInitializer`. If two test suites run in parallel (e.g., two Maven Surefire forks, or two CI jobs on the same machine), both try to bind embedded Redis to port 6381 — the second one fails.

**Fix:**
```java
// Let the OS pick an available port
redisServer = RedisServer.newRedisServer().build();  // uses random port
redisServer.start();
int port = redisServer.getBindPort();
TestPropertySourceUtils.addInlinedPropertiesToEnvironment(context,
    "spring.data.redis.port=" + port);
```

Or with Testcontainers (preferred for production CI):
```java
@Container
static GenericContainer<?> redis = new GenericContainer<>("redis:7-alpine")
    .withExposedPorts(6379);

@DynamicPropertySource
static void redisProperties(DynamicPropertyRegistry r) {
    r.add("spring.data.redis.port", () -> redis.getMappedPort(6379));
}
```
Testcontainers handles port allocation, Docker lifecycle, and container reuse automatically.

## Q8: "What is MDC, how does it work at a thread-pool level, and what is the risk?"

**A:** MDC is stored in a `ThreadLocal` map — one map per thread. In a thread-pool-based web server (Tomcat), threads are reused across requests. The risk:

```java
// Thread 1 processes request for user-001:
MDC.put("userId", "user-001");
// ... processes request ...
// FORGOT to call MDC.remove("userId") — maybe due to an exception bypassing cleanup

// Later, Thread 1 is reused for a request for user-002:
MDC.put("userId", "user-002");
// But the OLD value "user-001" might leak if the framework logs before MDC.put
```

This is why `MDC.remove("userId")` is in a `finally` block — it executes even if an exception is thrown, guaranteeing the `ThreadLocal` is cleared before the thread returns to the pool.

**Virtual threads (Java 21):** With `spring.threads.virtual.enabled=true`, each request gets a new virtual thread. MDC still uses `ThreadLocal` (now backed by the virtual thread's carrier thread), but since virtual threads are not pooled in the traditional sense, the leak window is smaller. However, structured concurrency in Java 21 introduces `ScopedValue` as a cleaner alternative to `ThreadLocal` for request-scoped data.

## Q9: "How would you add a `GET /balance/{userId}` endpoint without violating the architecture?"

**A:** The change follows the hexagonal layers:

1. **OpenAPI** — add the `GET /balance/{userId}` path to `openapi.yaml`. Run `mvn generate-sources` to generate the DTO and interface.

2. **Domain** — add `GetBalanceUseCase` interface (primary port) and `GetBalanceService` implementation in the `domain` module. `GetBalanceService` calls `accountSpi.findById(userId)` — no new SPI needed.

3. **App** — create `BalanceController implements BalanceApi` (or add the method to `PaymentController` if it logically belongs there). Inject `GetBalanceUseCase`. No business logic in the controller.

4. **Infra** — no changes. `AccountSpi.findById()` already exists.

5. **Bootstrap** — no changes. Spring auto-wires everything.

**What NOT to do:** Do not call `AccountRedisRepository` directly from the controller. Do not put Redis query logic in the controller. Do not skip the domain use-case "because it's just a read."

## Q10: "This code uses `BigDecimal.compareTo(BigDecimal.ZERO) <= 0` to validate amounts. Can you write a test that passes with a buggy `> 0` condition instead of `>= 0`?"

**A:** This is testing the boundary condition. The domain rejects `paymentAmount <= 0`:

```java
if (paymentAmount.compareTo(BigDecimal.ZERO) <= 0) {
    throw new InvalidPaymentAmountException(...);
}
```

If someone changed `<= 0` to `< 0` (buggy — allows zero payment):
```java
// This test would FAIL with the bug:
assertThatThrownBy(() -> useCase.process("user-1", BigDecimal.ZERO))
    .isInstanceOf(InvalidPaymentAmountException.class);
```

The existing test in `ProcessPaymentServiceTest` covers exactly this:
```java
@Test
@DisplayName("Zero paymentAmount → InvalidPaymentAmountException")
void process_zeroAmount() {
    assertThatThrownBy(() -> useCase.process("user-1", BigDecimal.ZERO))
        .isInstanceOf(InvalidPaymentAmountException.class);
    verifyNoInteractions(accountSpi);  // confirms no I/O was attempted
}
```

This is why boundary tests matter — `amount = 0` is the exact boundary. Any validator written as `amount < 0` (missing the equals) would pass all "negative amount" tests but fail the zero test.
